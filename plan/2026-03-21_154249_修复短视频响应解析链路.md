# 修复短视频响应解析链路

生成时间：2026-03-21 15:42:49
文件名：2026-03-21_154249_修复短视频响应解析链路.md

## 任务背景
用户要求分析项目中的短视频提示词请求功能，实际发起一次请求并查看模型返回结果，确认其是否符合程序要求。经验证，模型返回的内层 JSON 结构基本正确，但后端当前仅保留了 OpenAI 风格外层包装，未将 choices[0].message.content 提取并转换为程序可消费的 scripts 结构，导致前端无法识别和展示结果。

## 方案详情

### 现状分析
- 短视频请求入口位于 server/src/index.js 的 /api/products/:recordId/video-script 与 /api/products/:recordId/video-script/debug。
- equestShortVideoPromptScripts 当前在拿到 HTTP 200 后，仅将整个响应文本作为 raw passthrough 保存，没有进一步提取模型文本内容并解析业务 JSON。
- 
ormalizeShortVideoOutputPayload 期望接收业务 JSON，但当前传入为空对象，最终导致 outputPayload.scripts 为空。
- 调试接口即使请求成功，也会在原状态为 failed 时继续保留 failed，不利于判断本次调试是否成功产出可用结果。

### 技术方案
1. 在 equestShortVideoPromptScripts 中，从外层响应 JSON 中提取模型文本内容（优先使用已存在的 extractOpenAiLikeTextContent）。
2. 将提取出的文本再次解析为业务 JSON，并交给 
ormalizeShortVideoOutputPayload 生成标准化结构，同时保留原始响应与模型文本供调试展示。
3. 当调试请求成功且已得到可解析结构时，将调试状态更新为 completed，确保前端可以直接消费和展示结构化结果。
4. 使用已有前端展示逻辑验证 scriptSets / 原始响应信息是否同时可用，避免引入额外复杂度。

### 影响范围
- server/src/index.js
- 运行时保存到 server/data/products.json 的短视频结果结构
- 前端现有短视频展示逻辑（无需改 UI，依赖后端返回修复后自动生效）

## 原子步骤清单

### 步骤 1：固化响应解析入口
- **操作对象**：server/src/index.js
- **具体动作**：在短视频请求成功分支中补充模型文本提取与业务 JSON 解析流程
- **预期结果**：后端能从 OpenAI 风格包装响应中提取真正的短视频 JSON
- **关键里程碑**：是

### 步骤 2：标准化并落库短视频结果
- **操作对象**：server/src/index.js
- **具体动作**：将解析后的业务 JSON 传入标准化函数，填充 outputPayload.scripts 与 scriptSets
- **预期结果**：前端能够识别并展示短视频结构化结果
- **关键里程碑**：是

### 步骤 3：修正调试成功状态
- **操作对象**：server/src/index.js
- **具体动作**：调试接口成功时按本次结果更新状态，而不是机械保留 failed
- **预期结果**：调试成功可直观看到 completed 状态
- **关键里程碑**：否

### 步骤 4：执行回归验证
- **操作对象**：本地服务与短视频调试接口
- **具体动作**：再次发起短视频调试请求，核对原始响应、结构化结果、前端消费字段
- **预期结果**：确认“模型内层返回”和“程序实际可消费结果”一致
- **关键里程碑**：否

## 预期结果
- 短视频提示词接口能够正确解析模型外层包装并提取内层业务 JSON。
- outputPayload.scripts、scriptSets 等前端依赖字段被正确保存。
- 调试成功后状态与结果一致，便于继续排查内容层规则问题。
